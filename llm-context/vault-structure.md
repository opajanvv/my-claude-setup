# Vault structure

Reference for folder organization, frontmatter formats, and file handling.

Synced via Google Drive (no git history).

Wikilinks use Obsidian's shortest-path format: `[[filename]]` or `[[filename|display text]]`. Always verify the target file exists before creating a wikilink.

## Folder overview

- `planning/projects/` - What's on the workbench (committed work)
- `planning/ideas/` - The pin-up board (someday-maybes)
- `planning/tasks/inbox/` - Quick captures, unsorted
- `planning/tasks/active/` - Things being worked on
- `planning/templates/` - Obsidian templates for planning items
- `planning/TODAY.md` - Daily overview (generated by `planning-refresh`)
- `docs/` - Technical documentation
- `docs/templates/` - Obsidian templates for docs
- `llm-context/` - Summaries for Claude

## Tasks

Single actions - obligations, errands, quick jobs. Completed tasks get status: done, then archived by planning-refresh.

**Stale task review:** During process-inbox, tasks older than 3 weeks with no due date and no project link are flagged for review. Options: keep, demote to idea, or delete. This prevents the task list from growing indefinitely with wishful commitments.

### Frontmatter

```yaml
---
status: inbox | active | done | killed
priority: low | medium | high
project: optional-project-name
start: YYYY-MM-DD (optional, when the task becomes actionable)
due: YYYY-MM-DD (optional)
tags: [tag1, tag2]
created: YYYY-MM-DD
reviewed: YYYY-MM-DD (optional, set when kept during stale review)
---
```

### Start and due interaction

The `start` field creates an availability window. Categorization logic:
- `start` in the future → task is hidden (not actionable yet)
- `start` ≤ today, `due` within 7 days → stays in "This week" (no duplication with "Available")
- `start` ≤ today, `due` beyond 7 days or no due → appears in "Available"
- No `start` → behaves as before (categorized by due date or priority only)

### Setting start dates

Deduce start dates from context when creating tasks. Only ask when ambiguous.

Set `start` when:
- Jan names a specific date or timeframe ("not before March", "next week")
- The phrasing implies availability window ("any time this week" → start: today, due: end of week)
- There's an external dependency ("after the server migration")

Leave `start` empty when:
- Task can be done anytime with no urgency signal
- No timing clue in the request

### Inbox processing

Inbox items may be rough: just a title, typos, non-standard formats. Interpret intent, expand into proper tasks, ask Jan if unclear.

Steps:
1. Check planning root for stray files first (move to tasks/inbox/)
2. Fix typos, expand shorthand
3. Default destination is idea, not task. Only promote to task when there's a concrete trigger: due date, someone waiting, project link, or doing it this week
4. If promoting to task: set priority, add project link if relevant (only to active projects, never archived ones), add an energy tag (`quick`, `focus`, `deep`, `errand`) if the effort level is clear, move to `active/`
5. If an item references a file in a temporary location (Downloads, /tmp), copy/move it to the vault immediately

## Projects

Multi-step work on the workbench.

### Status flow

- `active` - On the workbench
- `paused` - Set aside, will return
- `done` - Finished
- `killed` - Abandoned, not worth completing

### Frontmatter

```yaml
---
status: active | paused | done | killed
target: true  # Optional: front and center on the bench
tags: []
created: YYYY-MM-DD
last_progress: YYYY-MM-DD  # Optional: updated when a step is checked off
completed: YYYY-MM-DD  # When done
---
```

### WIP limit

Active projects (target + active combined) are capped at 3. `planning-scan` computes `projects.wip_warning` with `exceeded: true` when over the limit. During `/process-inbox`, Claude walks through active projects and asks to pause or kill until under the limit before proceeding to focus selection.

### Stagnation

A project is stagnant if it has unchecked steps and `last_progress` (or `created` if `last_progress` is unset) is more than 21 days ago. Stagnant projects appear in TODAY.md "Needs attention" as `STALLED`. Update `last_progress` whenever a step is checked off.

### Completing a project

See `planning/CLAUDE.md` for the full checklist.

### Deferring scope

When something doesn't fit current scope: create a new project, link with wikilink ("deferred to [[new-project]]").

## Ideas

Notes pinned to the board. Not committed to yet.

### Status

- `someday` - Default, pinned to board
- `done` / `killed` - Archived by planning-refresh, purged after 4 weeks
- When ready to commit: move to `projects/`
- When no longer interesting: set status to killed

### Frontmatter

```yaml
---
status: someday
tags: []
created: YYYY-MM-DD
reviewed: YYYY-MM-DD  # After review
---
```

**Folders = primary category. Tags = cross-references.** Move files to correct folder rather than just tagging. Category subfolders are discoverable via `ls ideas/`.

## Docs

Technical documentation, primarily homelab. For structure and frontmatter formats, see `docs/homelab/CLAUDE.md`.

## TODAY.md

Generated by `planning-refresh` (jq template). See `planning/CLAUDE.md` for section overview. Tasks linked to projects are not shown in "needs attention" or "backlog" - they appear through their project. Low-priority tasks are tracked but not shown in TODAY.md.

## Tags

Energy/context tags (use on tasks to enable smart daily suggestions). Store without `@` in YAML — displayed as `@tag` in TODAY.md:
- `quick` - ≤15 min, no ramp-up needed
- `focus` - 30-60 min, needs concentration
- `deep` - 1-2h+ uninterrupted work
- `errand` - offline or physical action
